# AI ルール

本ドキュメントは、AYG プロジェクトにおいて AI が開発支援を行う際の最上位ルールである。
本ルールは他のすべてのルール・指示より優先する。
AI は本ルールに従わない操作を行ってはならない。

---

# 最上位ルール

## 1. 明示的承認なしに変更を実行しない

AI は、ユーザーから明示的な承認を得る前に、ファイル・設定・コード・Git・外部サービス・データ等へのいかなる変更も実行してはならない。
「進めてください」「実行してください」など、具体的な操作が承認された場合に限り実行する。
曖昧な指示や文脈からの推測だけでは実行してはならない。

## 2. 実行前承認が必要な操作の範囲

以下の操作は、すべて実行前にユーザーの承認が必要である。
例外はなく、小さな変更であっても事前承認を省略してはならない。

- ファイルの作成・編集・削除
- シェルコマンド、スクリプト、ビルドコマンドの実行
- パッケージ・依存関係の追加・更新・削除
- Git 操作（commit、branch、merge、rebase 等）
- GitHub 操作（push、PR 作成、Issue 操作等）
- データベースの変更（スキーマ、マイグレーション、データ投入・削除等）
- 外部サービスの変更（Supabase、API 設定、デプロイ設定等）

## 3. 実行前に提示する内容

承認を求める際、AI は必ず以下の 5 項目を提示しなければならない。

1. **実行目的** — なぜこの操作が必要か
2. **実行内容** — 具体的に何を行うか（コマンド、ファイル、手順）
3. **変更対象** — 影響を受けるファイル、ディレクトリ、サービス、データ
4. **想定される影響** — 正常完了時にどう変わるか
5. **リスクまたは戻し方** — 失敗時・副作用時の対処、ロールバック方法

上記が揃っていない状態で、実行を開始してはならない。

## 4. 承認された範囲のみを実行する

AI は、ユーザーが承認した範囲・内容・手順のみを実行する。
承認されていない追加作業、関連作業、改善、リファクタリングを勝手に行ってはならない。
「ついでに」「念のため」「一般的には必要なので」などの理由で作業範囲を広げてはならない。

### 4.1 承認の解釈

- 「A をしてください」→ A のみ実行可。B は不可。
- 「A と B をしてください」→ A と B のみ実行可。
- 承認されていないファイル・コマンド・設定変更は、たとえ有益であっても実行しない。

## 5. 実行後の報告

操作を実行した後、AI は必ず以下を報告しなければならない。

- **変更したファイル** — 作成・編集・削除したファイルの一覧
- **実行したコマンド** — 実行したコマンドとその目的
- **結果** — 成功・失敗、出力の要点、生成物の状態
- **エラー** — 発生したエラー、その原因、未解決の場合は次に必要な判断

報告なしに次の作業へ進んではならない。

## 6. 自動生成されるものを手作業で作らない

Flutter、Supabase CLI、各種ジェネレータ、ビルドツール等が自動生成するファイル・ディレクトリ・コードは、人間または AI が事前に手作業で作成してはならない。

例（手作業で作らないもの）:

- `flutter create` が生成する `lib/` 配下の標準構成
- ビルド成果物（`build/` 等）
- ツールが生成する `.g.dart`、`*.freezed.dart` 等
- マイグレーションや型定義の自動生成結果

必要な場合は、生成コマンドとその目的を提示し、承認後にコマンド実行で生成する。

## 7. 重要操作は個別の明示承認を得る

以下の操作は影響が大きいため、他の作業の承認に含まれていても、**個別に明示的な承認**を得なければならない。

- ファイル・ディレクトリの削除
- 既存ファイルの上書き（特に大量変更、設定ファイル、ドキュメント）
- DB スキーマの変更・マイグレーションの適用
- `git push` およびリモートへの反映
- 本番環境または外部公開に関わる変更
- 元に戻せない、または戻すコストが高い操作

「初期セットアップの一環」などの理由で、個別承認を省略してはならない。

## 8. 機密情報をコードや Git に保存しない

以下をコード、設定ファイル、ドキュメント、Git リポジトリに保存してはならない。

- API キー
- パスワード
- 秘密鍵（private key）
- 個人情報

機密情報は環境変数（`.env` 等）で管理し、`.env` 自体は Git 管理対象外とする。
既にコミットされた機密情報を発見した場合、ユーザーに報告し、削除・ローテーションの判断を仰ぐ。

## 9. 不明点・複数の選択肢がある場合は質問する

仕様、要件、実装方針、優先順位が不明な場合、AI は推測で進めてはならない。
複数の妥当な選択肢がある場合、選択肢とそれぞれのトレードオフを提示し、ユーザーに判断を求める。

推測で進めてよいケースは、ユーザーが明示的に「任せる」「おまかせ」と指示した場合のみとする。

## 10. プロジェクトドキュメントに反する変更を行わない

AI は、以下のドキュメントに記載された内容に反する変更を、ユーザーの明示的な承認なしに行ってはならない。

- `docs/01_PRD.md`（PRD）
- `docs/02_SDL.md`（SDL）
- `docs/05_DECISION_LOG.md`（決定ログ）

ドキュメントと実装・提案が矛盾する場合、矛盾点を指摘し、どちらを優先するかユーザーに確認する。
ドキュメントの更新が必要な場合も、更新内容を提示し承認を得てから行う。

## 11. 提案と実行を分離する

AI は「提案」と「実行」を明確に区別する。

ユーザーが提案を求めた場合は、提案のみを行う。
提案した内容は、ユーザーが明示的に承認するまで実行してはならない。
提案した内容を、そのまま実装・編集・コミット・設定変更へ進めてはならない。

## 12. ドキュメントも変更対象である

README、PRD、SDL、Decision Log、TODO 等のドキュメントも通常のコードと同様に変更対象とする。
ユーザーの承認なしに更新してはならない。

## 13. 一般論よりプロジェクト方針を優先する

一般的なベストプラクティスよりも、本プロジェクトで合意された方針を優先する。
一般論とプロジェクト方針が異なる場合は、その違いを説明し、ユーザーに判断を求める。
勝手に一般論へ修正してはならない。

## 14. AI は最小単位で作業する

一度に大きな変更を提案・実行しない。
可能な限り、小さく分割し、

- 承認
- 実行
- 報告

を繰り返す。
1 回の変更はレビュー可能な規模を維持する。

# 運用上の補足

## 承認の求め方

- 1 つの承認要求には、原則 1 つの目的・1 セットの操作を含める。
- 複数の独立した作業がある場合、分けて承認を求める。
- ユーザーが「文案だけ提示」「ファイルは触らない」と指示した場合、その指示を最優先する。

## 本ルールとの関係

- `rules/DEVELOPMENT_RULES.md` — 開発プロセス全般のルール
- `rules/CODING_RULES.md` — コーディング規約

上記ルールと本ルールが矛盾する場合、**本ドキュメント（AI_RULES.md）の最上位ルールを優先する**。
